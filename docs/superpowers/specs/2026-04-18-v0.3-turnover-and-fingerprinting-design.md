# CodeAssay v0.3 — Turnover Rate & Tier-2 Fingerprinting

**Status:** Draft.
**Date:** 2026-04-18.
**Depends on:** v0.2 merged to `main` (detection pipeline, `ai_commits.source`, `detection_confidence` column).

## Goal

Make the numbers credible. Ship two capabilities:

1. **Code Turnover Rate** as a first-class metric, with industry benchmarks bundled in the package so users can cite "we are Xth percentile" in board decks.
2. **Tier-2 per-author statistical fingerprinting** — detect AI-assisted commits from authors whose historical baseline the current commit diverges sharply from, catching AI usage that leaves no explicit marker.

Together these move CodeAssay from "a useful local CLI" to "a tool an engineering lead will cite to leadership" — the v0.3 tagline is "make the numbers credible."

## Non-goals

- DORA metrics / cycle-time correlation (scope creep toward dev-productivity dashboards; deferred).
- ML-based turnover forecasting.
- Public per-author leaderboards (social risk; per-author rollups proper come later with anonymization defaults).
- Automatic CI enforcement of turnover thresholds — we expose exit codes, but users wire them to CI themselves.

## Feature 1: Code Turnover Rate

### Metric definition

Given a lookback window `W` (days before HEAD) and a rewrite window `R` (days after each commit during which its lines are considered "at risk"):

- `lines_written(W)` — total lines added by commits whose date falls in the last `W` days.
- `lines_discarded(W, R)` — subset of `lines_written(W)` that are subsequently removed or overwritten by commits within `R` days of the original commit.
- `turnover_rate(W, R)` = `lines_discarded / lines_written`.

Two cohorts, computed independently:

- `turnover_rate_ai` — restricted to commits in `ai_commits` table.
- `turnover_rate_human` — restricted to commits NOT in `ai_commits`.

**Headline metric:**

`turnover_ratio` = `turnover_rate_ai / turnover_rate_human`.

Interpretation: "your AI turnover is 2.3× your human turnover; industry-healthy is <2×."

Defaults: `W = 90` days, `R = 30` days. Both configurable in `.codeassay.toml`.

### Algorithm

For each commit C added in window W, for each file F modified by C:

1. Use `git show --numstat` to get lines added by C in F.
2. Use `git blame -L` or line-level blame at `HEAD` (or `C + R days`, whichever is earlier) to determine which of C's lines survive verbatim.
3. `discarded_lines = lines_added_by_C - surviving_lines_from_C_in_F`.

Cache blame results by `(commit_sha, file)` tuple in a new table; blame is the expensive call and v0.4's incremental-scan work will reuse this cache.

Aggregation is straightforward SQL over the new table.

### Data model

```sql
CREATE TABLE IF NOT EXISTS commit_lines (
    commit_sha TEXT NOT NULL,
    repo_path TEXT NOT NULL,
    file TEXT NOT NULL,
    lines_added INTEGER NOT NULL,
    lines_survived INTEGER NOT NULL,  -- in HEAD or C+R
    measurement_window_end TEXT NOT NULL,  -- ISO date
    PRIMARY KEY (commit_sha, repo_path, file)
);
```

Recomputed incrementally (same cadence as the scanner). Legacy DBs get the table via idempotent `CREATE TABLE IF NOT EXISTS`.

### Benchmarks

Ship `src/codeassay/benchmarks.json` in the package:

```json
{
  "turnover_rate": {
    "pre_ai_baseline": 0.033,
    "industry_2026": 0.057,
    "healthy_target": 0.04,
    "sources": [
      {"name": "Larridin Code Turnover Report 2026", "url": "…"},
      {"name": "GitClear Code Churn Research", "url": "…"}
    ],
    "last_updated": "2026-04-18"
  }
}
```

- Loaded via `importlib.resources.files("codeassay")` (same pattern as profiles).
- Refreshed on release cadence; one annual commit is enough.
- Dashboard uses the benchmarks to render a percentile-style card ("you sit at the 62nd percentile for AI turnover").

### Configuration

Add to `.codeassay.toml`:

```toml
[turnover]
lookback_days = 90           # window W
rewrite_window_days = 30     # window R
yellow_threshold = 0.04      # warn when above
red_threshold = 0.06         # fail CI when above (via --fail-on)
```

All fields optional; defaults above.

### CLI

- `codeassay scan` automatically computes turnover (no new flag needed).
- `codeassay report` adds turnover cards to both CLI and markdown output.
- New flag `codeassay scan --fail-on=turnover-red` — exits `1` if `turnover_rate_ai` exceeds `red_threshold`. Useful for CI gating.
- `codeassay export --format json` includes `turnover` section with `{ai, human, ratio, benchmarks_percentile}`.

### Dashboard

Two new elements:

1. **Metric card strip:** AI turnover %, human turnover %, ratio, percentile vs industry. Coloring by yellow/red thresholds.
2. **Trend chart:** stacked line of AI vs human turnover rate per month over the scan window. Benchmarks rendered as horizontal guide lines.

### Testing

- Unit: fabricated blame fixtures with known add/discard patterns → assert exact counts.
- Integration: `tests/test_turnover_integration.py` — small real repo with scripted commits (some AI-trailered, some human), AI discards its own lines (simulating rework), assert ratio is within expected band.
- Benchmark loader test: `benchmarks.json` parses, has required keys, values are `float` and between 0 and 1.
- Threshold test: `--fail-on=turnover-red` exits 1 when threshold breached, 0 otherwise.

## Feature 2: Tier-2 Per-Author Statistical Fingerprinting

### Goal

Catch AI-assisted commits that carry no explicit marker, by comparing each commit to the author's historical baseline. Inserted into the detection pipeline between profiles and the probabilistic scorer.

### Evaluation position in pipeline

Current order (from v0.2's `classify()`):

1. User rules
2. Built-in profiles
3. Probabilistic scorer (if enabled)

v0.3 order:

1. User rules
2. Built-in profiles
3. **Per-author fingerprint** (if author has ≥ `min_prior_commits` history)
4. Probabilistic scorer (if enabled)

First match wins. Fingerprinting is independent of scorer — enabling fingerprinting alone, without the scorer, is a supported configuration.

### Per-author baseline

For each author (keyed by normalized `author_email`), compute over all their commits in the repo prior to a target commit:

- `avg_diff_size` — mean lines added per commit.
- `comment_ratio` — fraction of added lines matching a language-agnostic comment regex (`^\s*(//|#|/\*|\*)`).
- `identifier_entropy` — Shannon entropy of variable/function/class names extracted from added lines. For v0.3, use a regex-based tokenizer (`[A-Za-z_][A-Za-z0-9_]{2,}`); tree-sitter is future work.
- `punctuation_density` — ratio of `.,;:!?` chars in the commit message.
- `message_length_chars` — mean length of commit messages.

Baselines update incrementally as `scan_repo` walks chronologically.

### Divergence check

For a target commit C by author A:

1. If A has fewer than `min_prior_commits` (default 20) commits before C, **skip** — fingerprinting is not yet valid.
2. Compute each of the 5 metrics for C.
3. For each metric, compute the Z-score relative to A's baseline (mean, stddev).
4. Count `divergent_metrics` = metrics with `|Z| >= sigma_threshold` (default 2.0).
5. If `divergent_metrics >= min_divergent_metrics` (default 3 of 5), flag as AI.

On flag, return `Detection(tool="unknown", confidence="medium", method="fingerprint", source=f"fingerprint:{author}:{divergent_metrics}/5", detection_confidence=<Z-score-derived, capped 50–85>)`.

### Data model

```sql
CREATE TABLE IF NOT EXISTS author_baselines (
    repo_path TEXT NOT NULL,
    author_email TEXT NOT NULL,
    metric_name TEXT NOT NULL,
    mean_value REAL NOT NULL,
    stddev_value REAL NOT NULL,
    sample_size INTEGER NOT NULL,
    last_updated_sha TEXT NOT NULL,
    PRIMARY KEY (repo_path, author_email, metric_name)
);
```

Small in absolute terms — a team of 10 devs × 5 metrics = 50 rows per repo.

### Configuration

Add to `.codeassay.toml`:

```toml
[fingerprint]
enabled = true
min_prior_commits = 20
sigma_threshold = 2.0
min_divergent_metrics = 3
```

Default: `enabled = true`. This is the first signal enabled by default that can "miss" (a human having a weird day could trigger), so the README must explicitly frame confidence and show how to tune or disable.

### CLI impact

- `codeassay detect-test <hash>` extended: when fingerprinting fires, print each metric's Z-score and which crossed the threshold. Makes debugging false positives tractable.
- No new subcommand. Existing `commits --source 'fingerprint:*'` already works.

### Testing

- Unit per metric: known-input → expected-output (e.g. a commit of 100 lines with 30 comment lines → `comment_ratio == 0.3`).
- Baseline math test: inject a synthetic author with 30 known commits → baseline mean/stddev match hand-calculated values.
- Divergence test: synthetic author baseline + a commit 3σ above mean on 3 metrics → flagged. Same but 2σ → not flagged.
- Integration: real-ish repo where one author has a consistent style for 25 commits then commits a conspicuously AI-styled block → fingerprint catches it; control commit by same author does not.

## Migration, back-compat, rollout

- All new tables use `CREATE TABLE IF NOT EXISTS`. No breaking DB changes.
- Turnover is computed always (no opt-out). Adds scan time proportional to blame calls.
- Fingerprinting is enabled by default. Users who see false positives can disable via config.
- The `source = "fingerprint:..."` string is a new value downstream code may not expect. Dashboard and reports must handle the `fingerprint:` prefix alongside `profile:`, `user:`, `score:`.

## Dependencies

- **No new hard runtime deps.** Regex-based identifier tokenization keeps us pure-Python.
- Optional future work: tree-sitter-based tokenization for better identifier extraction accuracy. Deferred.

## Implementation effort

~2–3 weeks of focused sessions, sequenced:

2. Benchmarks.json + dashboard cards + report cards (~2 days).

## Open questions

1. **Baseline recomputation frequency.** Update on every scan, or only when author has gained N new commits since last update? Recommendation: every scan; the math is O(commits) and cheap.
2. **Cross-repo baselines.** If the same `author_email` appears in multiple scanned repos, should baselines merge? Recommendation: per-repo only for v0.3. Cross-repo merging is a v0.6 feature alongside per-team rollups.
3. **Tree-sitter vs regex for identifier extraction.** Regex is simpler and dep-free but weaker on multi-language repos. Recommendation: regex for v0.3, re-evaluate at v0.5.
4. **Turnover window overlap with scan date.** If HEAD is older than `rewrite_window_days`, turnover for recent commits is undercomputed (lines not yet at risk long enough). Recommendation: emit a warning in the report if `W + R > days_since_first_commit`; don't silently misreport.

## Non-negotiables

- Local-first: no network calls for benchmarks lookup (the file ships in the package).
- No author names in public outputs unless the user opts in. Default reports show aggregate turnover; per-author views require an explicit command flag (for v0.6 when per-author rollups formalize).
- All new fields are SQL-queryable. The "drop down to SQL" escape hatch is preserved.

## Minor UX fixes to bundle into v0.3

Small items surfaced during v0.2 manual testing — not feature work, but cheap to ship alongside v0.3 while `config_init.py` is being touched anyway.

### Template appends break with duplicate-table TOML errors

**Symptom.** The `STARTER_TEMPLATE` emitted by `codeassay config init` declares each built-in profile as a full `[profiles.X]` table block:

```toml
[profiles.cursor]
enabled = true
```

A user who wants to disable a profile naturally appends at the bottom of the file:

```toml
[profiles.cursor]
enabled = false
```

TOML rejects this with `TOMLDecodeError: Cannot declare ('profiles', 'cursor') twice`. The correct edit is to change `enabled = true` → `enabled = false` on the existing line, but that is non-obvious to someone unfamiliar with TOML's table-declaration rules.

**Fix options:**

1. **Inline form.** Emit each profile as an inline table so users can tweak in one line without declaring a table block:
   ```toml
   profiles.cursor = { enabled = true }
   profiles.aider  = { enabled = true }
   ```
   Downside: less visually obvious that there is a `[profiles]` section.

2. **Keep block form, add guidance comment.** Add a comment above the profiles block:
   ```toml
   # To disable a profile: change `enabled = true` to `enabled = false`
   # on its line below. Do NOT add a second [profiles.<name>] block —
   # TOML forbids duplicate table declarations.
   ```

**Recommendation.** Option 2. Block form lines up visually with how `[[detect.author]]` arrays are shown below; switching to inline creates two styles in the same file. A three-line comment is enough and educates users about TOML.

**Scope.** One edit to `src/codeassay/detection/config_init.py`. No code changes, no new tests. Ship alongside the v0.3 turnover work.
